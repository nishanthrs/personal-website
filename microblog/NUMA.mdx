I was reading about Kubernetes workload scheduling and came across topology-aware scheduling. What the hell is topology?
How does it factor into how workloads are scheduled onto nodes? Apparently it's especially important for GPU-based workloads?

It turns out that modern CPUs divide their resources into NUMA nodes. Each NUMA node has its own CPU cores, local RAM,
and PCIe bus to network and disk and GPUs. NUMA nodes are connected to each other via CPU interconnects.
This topology prevents traffic jams in memory bandwidth, but a CPU in NUMA node #1 accessing memory in NUMA node #2
is much slower (3x slower in AMD EPYC CPUs). **Therefore, for high-performance and low-latency workloads, it's better to
schedule an entire workload onto one NUMA node.** For GPU workloads, the CPUs and GPU resources must be co-located on the
same NUDA node for maximum throughput and lowest latency.

![CPUs and GPUs NUMA Topology](/numa_topology.png)

Kubernetes has to offer topology-aware scheduling so that its workload scheduler can take NUMA nodes into account when
deciding which resources to assign to a workload. Otherwise, it can schedule CPU cores on different NUMA nodes or its
CPU cores on one NUMA node and its GPUs on another NUMA node, significantly slowing down the access time of RAM or the
latency of CPU assigning tasks to the GPU.
